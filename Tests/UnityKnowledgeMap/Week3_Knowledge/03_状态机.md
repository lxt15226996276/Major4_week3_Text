## 一、有限状态机（FSM）定义（官方标准 · 必背）

### 1.1 什么是状态机

**状态机（State Machine）**：通过定义**状态**及**切换规则**，管理对象在不同条件下的行为。

**有限状态机（Finite State Machine，FSM）**：状态集合**有限**，且任意时刻对象**只处于其中一个状态**。

```
        ┌─────────┐  条件：有输入   ┌─────────┐
        │  Idle   │──────────────→│ Moving  │
        └─────────┘←──────────────└─────────┘
              ↑      条件：无输入
              │
              │ 条件：攻击键
              ↓
        ┌───────────┐  条件：攻击结束
        │ Attacking │────────────────→ Idle
        └───────────┘
```

### 1.2 FSM 五要素（必背）

| 要素 | 英文 | 含义 | 频率 |
|------|------|------|------|
| **状态集合** | States | 所有可能行为模式（有限个） | ⭐⭐⭐ |
| **转移集合** | Transitions | 状态之间的有向切换 | ⭐⭐⭐ |
| **初始状态** | Initial State | 启动时进入的状态 | ⭐⭐⭐ |
| **当前状态** | Current State | 运行时唯一活跃状态 | ⭐⭐⭐ |
| **转移条件** | Conditions / Events | 何时允许 A → B | ⭐⭐⭐ |

**[重难点] FSM 核心约束**：任意时刻 **Current State 有且只有一个**。这是 FSM 与「多个 bool 标记」的本质区别。

### 1.3 状态机的优势（考纲 175）

| 优势 | 说明 |
|------|------|
| **逻辑清晰** | 每个状态职责单一，读 switch 即知行为 |
| **模块化** | 新增状态 = 加 enum 值 + case，少改旧代码 |
| **易维护** | 转移关系可画成状态图，团队共识 |
| **减少冗余** | 避免 if-else 嵌套地狱 |
| **易测试** | 单状态逻辑可独立测 |

**[易错点]** 不是状态越多越要状态机——2 个状态（Idle/Attack）就值得用 enum；1 个 bool 能搞定的不必硬上类层次。

## 二、FSM 形式化描述（⭐⭐⭐ · 理论深入）

### 2.1 数学表示

```
FSM = (S, Σ, δ, s₀, F)

S   = 有限状态集合     例：{ Idle, Moving, Attacking }
Σ   = 输入/事件字母表   例：{ HasInput, NoInput, AttackBtn, AttackEnd }
δ   = 转移函数         δ(Idle, HasInput) = Moving
s₀  = 初始状态         s₀ = Idle
F   = 终结状态集合（可选） 例：{ Dead }
```

### 2.2 转移三元组

每条转移可记为：

```
(源状态 Source, 条件 Condition, 目标状态 Target)
```

| 示例 | 含义 |
|------|------|
| `(Idle, HasInput, Moving)` | 待机时有输入 → 移动 |
| `(Moving, NoInput, Idle)` | 移动时无输入 → 待机 |
| `(Any, AttackBtn, Attacking)` | **全局过渡**：任意状态按攻击 → 攻击 |

### 2.3 状态生命周期（三回调）

| 回调 | 时机 | 典型用途 | 频率 |
|------|------|----------|------|
| **OnEnter** | 刚进入该状态 | 初始化计时器、播动画 Trigger | ⭐⭐⭐ |
| **OnUpdate** | 处于该状态每帧 | 移动、检测转移条件 | ⭐⭐⭐ |
| **OnExit** | 即将离开该状态 | 停协程、清标志、停音效 | ⭐⭐⭐ |

**[重难点]** 转移不是瞬间跳 case——完整 FSM 应先 **Exit 旧状态 → 改 Current → Enter 新状态**。

```
ChangeState(Moving):
  1. currentState.OnExit()
  2. currentState = Moving
  3. currentState.OnEnter()
```

**[易错点]** 只有 switch 没有 Enter/Exit → Attack 状态切走时协程还在跑、计时器残留。

## 三、enum + switch 实现（Lab P-L1 · ⭐⭐⭐）

Lab_Gameplay_01 最低标准：

```csharp
public enum PlayerState { Idle, Moving, Attacking }

public class PlayerStateMachine : MonoBehaviour
{
    PlayerState currentState = PlayerState.Idle;
    float attackTimer;

    void Update()
    {
        switch (currentState)
        {
            case PlayerState.Idle:
                UpdateIdle();
                break;
            case PlayerState.Moving:
                UpdateMoving();
                break;
            case PlayerState.Attacking:
                UpdateAttacking();
                break;
        }
    }

    void UpdateIdle()
    {
        Vector2 input = GetMoveInput();
        if (input.sqrMagnitude > 0.01f)
            ChangeState(PlayerState.Moving);

        if (Input.GetButtonDown("Fire1"))
            ChangeState(PlayerState.Attacking);
    }

    void UpdateMoving()
    {
        Vector2 input = GetMoveInput();
        if (input.sqrMagnitude < 0.01f)
            ChangeState(PlayerState.Idle);
        else
            DoMove(input);

        if (Input.GetButtonDown("Fire1"))
            ChangeState(PlayerState.Attacking);
    }

    void UpdateAttacking()
    {
        attackTimer -= Time.deltaTime;
        if (attackTimer <= 0f)
            ChangeState(PlayerState.Idle);
        // 攻击中不响应移动
    }

    void ChangeState(PlayerState newState)
    {
        if (currentState == newState) return;
        OnExitState(currentState);
        currentState = newState;
        OnEnterState(newState);
    }

    void OnEnterState(PlayerState state)
    {
        switch (state)
        {
            case PlayerState.Attacking:
                attackTimer = 0.5f;
                animator.SetTrigger("Attack");
                break;
        }
    }

    void OnExitState(PlayerState state) { }
}
```

### 3.1 禁止反面模式

```csharp
// ❌ P-L1 反面：多 bool 互斥
bool isIdle = true;
bool isMoving = false;
bool isAttack = false;
// 可能写错导致 isIdle && isMoving 同时为 true
```

| 写法 | 问题 |
|------|------|
| 多个 bool | 可同时 true/false，非 FSM |
| 散落 `currentState = X` | 绕过 Enter/Exit，难维护 |
| 每状态一个 MonoBehaviour | Update 顺序不可控 |

**[教材精华·Lab]** 用 **一个** `switch (currentState)` 驱动；状态用 **枚举 + switch** 或 **简单类**，不用 5 个 bool 互斥。

## 四、状态类 + 入口类模式（⭐⭐⭐）

P+1 / 工程化升级：每个状态一个类，统一接口。

```csharp
public interface IState
{
    void Enter();
    void Update();
    void Exit();
}

public class IdleState : IState
{
    readonly PlayerContext ctx;
    public IdleState(PlayerContext ctx) => this.ctx = ctx;

    public void Enter() { }
    public void Update()
    {
        if (ctx.HasMoveInput()) ctx.FSM.ChangeState(ctx.MovingState);
        if (ctx.AttackPressed()) ctx.FSM.ChangeState(ctx.AttackState);
    }
    public void Exit() { }
}

public class StateMachine
{
    IState current;
    public void Init(IState start) { ChangeState(start); }

    public void ChangeState(IState next)
    {
        current?.Exit();
        current = next;
        current.Enter();
    }

    public void Update() => current?.Update();
}
```

**[教材精华·Nystrom/Gregory]** State 模式：把每个状态的 Update 封装进类，switch 移到多态 `Update()`，新增状态不改 StateMachine 核心。

| 方案 | 适用 |
|------|------|
| enum + switch | Lab、2～4 状态、学习 FSM |
| 状态类 + 入口类 | 5+ 状态、AI、需复用 |

## 五、如何定义状态切换关系（考纲 169 · ⭐⭐⭐）

### 5.1 转移表（推荐设计方式）

先画表再写代码：

| 源状态 | 条件 | 目标状态 |
|--------|------|----------|
| Idle | 有移动输入 | Moving |
| Moving | 无输入 | Idle |
| Idle / Moving | 攻击键 | Attacking |
| Attacking | 攻击计时结束 | Idle |
| **Any** | HP ≤ 0 | Dead |

### 5.2 转移检查顺序

```
每帧 Update：
  1. 先检查全局过渡（Die、Stun）— 优先级最高
  2. 再检查当前状态的局部转移
  3. 若多条条件同时满足 → 设计互斥或优先级
```

**[易错点]** Chase 和 Attack 距离条件重叠 → 需 `attackRange < chaseRange`，进 Attack 后不再 Chase。

## 六、全局过渡（考纲 173 · ⭐⭐⭐）

**全局过渡**：从**任意**当前状态都能切换到目标状态的规则。

| 典型 | 说明 |
|------|------|
| **Die** | 任何状态 → 死亡 |
| **Stun** | 任何状态 → 硬直 |
| **Pause** | 任何状态 → 暂停 |

```csharp
void Update()
{
    // 全局过渡 — 放在 switch 之前
    if (hp <= 0)
    {
        ChangeState(PlayerState.Dead);
        return;
    }

    switch (currentState) { ... }
}
```

**[对比 Animator]** Mecanim 的 **Any State** → 玩法 FSM 的 **全局过渡**；见 [Week2/03_动画状态机一.md](../Week2_Animation/03_动画状态机一.md)。

**[糟粕剔除]** 每个 case 里复制粘贴 `if (hp <= 0) Dead` → 应提为全局过渡。

## 七、延迟切换（考纲 172 · ⭐⭐）

**延迟切换**：条件满足后**不立即**转移，等计时器到期再切。

```csharp
float stunTimer;

void UpdateStunned()
{
    stunTimer -= Time.deltaTime;
    if (stunTimer <= 0f && IsGrounded())
        ChangeState(PlayerState.Idle);
}
```

| 场景 | 做法 |
|------|------|
| 攻击后摇 | Attacking 状态内 timer，结束才回 Idle |
| 受击硬直 | Enter Stunned 设 timer = 0.3s |
| 技能 CD | Exit Attack 时记录 `lastAttackTime` |

**[考纲精简]** OnUpdate 中加计时器，达标且条件满足时切换。

## 八、层次状态机：超状态与子状态（考纲 171 · ⭐⭐）

### 8.1 概念

| 概念 | 说明 |
|------|------|
| **超状态（Super State）** | 包含多个子状态的「大状态」 |
| **子状态（Sub State）** | 超状态内部的具体状态 |

```
Super: Combat（战斗模块）
├── Sub: Attack
├── Sub: Block
└── Sub: Dodge

Super: Locomotion（ locomotion 模块）
├── Sub: Idle
├── Sub: Walk
└── Sub: Run
```

### 8.2 作用

| 作用 | 说明 |
|------|------|
| **简化结构** | 相关状态归类，主状态图不爆炸 |
| **共享逻辑** | 超状态 Enter：拔武器；Exit：收武器 |
| **模块化** | 战斗/移动/交互各自一个子状态机 |

**[教材精华·宣雨松/Horachek]** 子状态机 = 玩法 FSM 的「Nested Sub-State Machine」，与 Animator Sub-State Machine **概念同构、实现分离**。

### 8.3 代码示意

```csharp
// 外层：Locomotion / Combat
// 内层：Combat 下再 enum { Attack, Block }
StateMachine locomotionSM;
StateMachine combatSM;  // 仅 Enter Combat 超状态时激活
```

**[频率]** Lab P-L1 不要求层次 FSM；Lab_AI_01 四状态用 flat enum 即可；P+1 可口述超状态。

## 九、并行状态机 / 并发状态（考纲 168 · ⭐⭐）

**并行状态机**：允许多个状态**同时**运行——严格说已超出经典 FSM，是 **FSM 的扩展**。

| 例子 | 并行状态 |
|------|----------|
| 角色 | **Locomotion=Moving** + **Buff=Poisoned** |
| 玩家 | **Body=Attack** + **UpperBody=Aiming**（常与 Animator Layer 配合） |

```
         ┌─ Locomotion FSM ─ Idle / Move / Jump
Actor ───┤
         └─ Status FSM ─ Normal / Stunned / Poisoned
```

**[重难点]**
- **互斥 FSM**：同一层只能一个状态（Lab 默认）
- **并行 FSM**：多层各跑各的，层间用接口通信

**[面试·Lab]** 玩法状态机和 Animator 状态机能合并吗？
→ **可协同、不建议合并为一个**。玩法 FSM 管逻辑；Animator 管动画。通过 `SetBool/SetTrigger` 同步。Animator Layer 可实现动画层并行，类似并行 FSM。

| 类型 | 管什么 | 实现位置 | 文档 |

### 协同关系

```
玩法 FSM（Attacking）
    ↓ SetTrigger("Attack")
Animator 状态机（Attack State → 播 Clip）
    ↓ Animation Event
玩法 FSM（OnAttackHit 伤害帧）
```

**[重难点]** 脚本职责分工：
- 玩法 FSM：**能不能**攻击、**何时**切状态
- Animator：**播放哪个**动画、**如何**混合过渡

**[糟粕剔除]** 在玩法 switch 里 `animator.Play("Run")` 绕过 Animator Transition → 与 Week2 规范冲突；应 Set 参数。

## 十三、与 Lab / Exam 衔接（⭐⭐⭐）

### Lab_Gameplay_01（U13）

| 要求 | 实现 |
|------|------|
| 至少 2 状态 | Idle + Attacking，或 + Moving |
| enum + switch | `PlayerState` |
| 攻击中不能移动 | Attacking 的 Update 不调用 Move |
| 与摇杆 | 见 [02_摇杆插件.md](02_摇杆插件.md) |

### Lab_AI_01 预告（U14 · 见 04 文档）

| 状态 | 转移条件 |
|------|----------|
| Patrol | 默认 |
| Chase | 距离 < chaseRange |
| Attack | 距离 < attackRange |
| Return | 距离 > loseRange |

→ 同一套 FSM 框架，状态数从 3 扩到 4。

### Exam07/08

主场景移动可复用 `PlayerStateMachine` 结构。

## 十四、注意要点与最佳实践

### 常见陷阱

| 易错点 | 说明 |
|--------|------|
| 多 bool 互斥 | 违反 FSM 定义 |
| 无 ChangeState 入口 | Enter/Exit 遗漏 |
| 转移条件重叠 | 同时满足 Attack 和 Chase |
| 玩法与动画合并 | 逻辑动画耦死 |
| 全局过渡复制 N 份 | 应提至 switch 前 |
| Find 玩家每帧 | Awake 缓存 Transform |

### 最佳实践

- 先画**转移表/状态图**，再写代码
- 统一 `ChangeState()`，内含 Exit → 赋值 → Enter
- 全局过渡（Die/Stun）最高优先级
- 玩法 FSM 驱动 Animator 参数，不直接 Play Clip
- 从 enum + switch 起步，状态 >5 考虑状态类
- Debug.Log 转移链，便于 Lab 测试
